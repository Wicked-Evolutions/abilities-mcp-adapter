> **Status:** v1 final, ratified 2026-05-07. Co-authored by GPT 5.5 + Claude Code orchestrator.

## North Star

The official WordPress Abilities API is the platform contract. Our suite is product layers around that contract — not a replacement for it.

| Layer | What it does | What it does NOT do |
|---|---|---|
| `abilities-for-ai`, `abilities-for-fluent-plugins`, etc. | Register WordPress abilities via `wp_register_ability()` | Define a parallel registry |
| `abilities-mcp-adapter` (MCP Adapter) | Project, scope, redact, rate-limit, transport | Silently invent a different contract |
| `abilities-mcp` (Bridge) | Connect AI clients to the projected surface | Replace or override registry behavior |

A single sentence rule: **productize around the registry, not instead of it.**

Product layers may add operator controls around discovery and execution, but those controls must be transparent, documented, and testable as layers above the registered ability.

## Principles (binding)

### 1. Registry First

Every product or plugin capability is registered as a real WordPress ability — `wp_register_ability()` with namespaced name, category, `input_schema`, `output_schema`, `execute_callback`, `permission_callback` — unless a documented exception with reasoning exists in the relevant repo.

Ability categories are registered on `wp_abilities_api_categories_init`; abilities are registered on `wp_abilities_api_init`. Any compatibility shim must preserve those lifecycle semantics so discovery is consistent across REST, the adapter, CLI, and future official AI clients.

### 2. Native APIs Over Raw Storage

Ability callbacks use WordPress core APIs and parent-plugin public APIs before direct database writes or option mutation. Raw SQL, direct `update_option()` mutation outside intended scope, and filesystem writes require a documented reason, capability check, and cleanup/rollback story.

This protects audit trails, caches, hooks, webhooks, and plugin invariants that native APIs maintain. It also makes our third-party ability plugins (Abilities For Fluent and similar) safe reference implementations rather than bypass wrappers around tables.

### 3. Adapter Is A Projection

Adapter discovery, dispatch, and metadata project the WordPress registry. The adapter may filter, redact, annotate, scope, and rate-limit. The adapter may NOT redefine schemas, override permission semantics of registered abilities, or invent alternate versions of WordPress/plugin business abilities that bypass the registry.

The adapter may register adapter-control abilities in its own namespace — discovery (`mcp-adapter-discover-abilities`), `mcp-adapter-get-started`, `mcp-adapter-execute-ability`, redaction settings, health checks, and similar transport/control-plane operations. These are not WordPress business-domain abilities; they are control plane. The distinction is enforced by namespace: business-domain abilities remain registered by the product/plugin that owns the domain behavior.

### 4. Schemas Stay WordPress-Native

JSON Schema is the source of truth. Use WordPress and plugin-domain names: `post_id`, `term_id`, `meta_key`, `option_name`, etc. Do NOT force ability inputs into an invented universal convention. There is no generic `id`/`name`/`value` rename pass.

The official ability schema contract is the WordPress Abilities API schema contract. Stricter client-specific profiles — Anthropic/OpenAI tool-schema constraints, etc. — are adapter or projection concerns unless WordPress core adopts them. Do not "fix" WordPress-native schemas at the source layer to satisfy a single AI client.

### 5. Permissions Stay Layered, Not Replaced

WordPress `current_user_can()` is the base truth. OAuth scopes and module-level permission toggles are *additional* gates that compose on top — they never substitute for WP capability checks.

Diagnostics must identify which layer blocked execution: WordPress capability, OAuth scope, module toggle, per-ability override, license/tier gate, or module/plugin state. Operator and AI-client UX depend on knowing which gate fired, not just that "permission was denied."

### 6. Errors Stay Compatible

Native ability callbacks return data or `WP_Error`. MCP/JSON-RPC envelopes are produced at the adapter boundary and must not leak backward into ability callback contracts.

The adapter may normalize to MCP envelope shape at the response boundary. The underlying ability still behaves like WordPress to any caller using REST, PHP, or future official AI tooling.

### 7. Metadata Extensions Are Namespaced

Custom metadata uses namespaced keys inside official extension locations — for example `meta['wicked_evolutions']`, `meta['mcp']`, or namespaced annotation keys. Do NOT redefine official top-level fields such as `name`, `category`, `input_schema`, `output_schema`, `execute_callback`, or `permission_callback`.

Use nested array structures in PHP rather than literal dotted-string keys; the keys must reflect the actual stored shape, not a string-key fiction.

### 8. Official Interface Parity

For every public business-domain ability, the official WordPress Abilities API path (`GET|POST|DELETE /wp-abilities/v1/abilities/{name}/run`) and the MCP adapter path preserve equivalent native execution semantics for the same authenticated user and input.

Intentional adapter-layer differences — OAuth scope denial, redaction, MCP envelopes, rate limits, transport/session errors, tool-name projection — must be documented and tested as adapter-layer differences, not as silent divergence from native semantics.

### 9. Scope Coverage Is Derived Or Coverage-Tested

OAuth scopes are derived from registered ability categories or validated by automated coverage tests.

A CI test compares the registered public ability categories visible in fixtures or live catalog snapshots against the OAuth scope registry. Every category must be grantable, intentionally mapped, or explicitly marked non-OAuth-executable. New ability categories cannot ship without scope mapping. Missing-scope incidents (adapter #101 `site`, #102 `surecart-ecommerce`) cannot recur.

### 10. Stable Ability Contracts

Ability names, input schemas, output schemas, categories, and permission semantics are public contracts once released. Breaking changes require deprecation, aliasing, changelog notes, and a named removal version.

This principle prevents another registry-wide rename sprint from sneaking back in under "polish."

### 11. WPCS Is CI Hygiene, Not Sprint Scope

WordPress Coding Standards run in CI. Failures caused by touched code block merge. WPCS keeps PHP shape WordPress-native and contributor-familiar.

Do NOT use WPCS as a vehicle to expand sprint scope or block alpha on formatting churn. Repo-wide legacy formatting debt is handled in separate hygiene PRs, not folded into feature work.

## Decision Heuristic

Before merging any non-trivial change, ask in order:

1. Does this move us closer to the WordPress ability registry being the source of truth?
2. Would this change still make sense if WordPress core, not our adapter, were executing the ability?
3. Is this a product layer, or are we accidentally redefining the platform?
4. Could another plugin author copy this pattern and still feel they're building WordPress-native abilities?
5. Are we using the native WordPress/plugin API path, or are we bypassing it for convenience?

If any answer is unclear, pause and design-review against this doc before merging.

## Drift Examples — what we don't do

Grounding for the principles. Living record — appended as new violations are caught and corrected.

| Proposal | Violated | Why it drifted |
|---|---|---|
| Force `post_id` → `id` and `meta_key` → `name` registry-wide rename | Principle 4 | "Universal convention" rewriting WordPress-native names |
| Replace per-ability `WP_Error` with a unified MCP envelope at the ability layer | Principle 6 | Adapter shape leaking into ability callback contract |
| `automation_mode` flag in OAuth metadata to suppress Knowledge Layer directive | Principle 3 | Bending OAuth protocol to carry adapter-specific behavior |
| Make `patterns/register` imply persistent WP-native registration when WP only registers runtime patterns | Principles 1 & 3 | Product persistence pretending to be native registry behavior (a clearly-named managed `patterns/save-managed-pattern` would be valid) |
| Manually adding scope strings to `ScopeRegistry` for each new ability category | Principle 9 | Drift-prone vs. derived/coverage-tested |

Caught during contract-polish review against this doc's draft. Future drifts get caught at PR or sprint-plan review using the same lens.
